# 产品负责人角色章程

## 使命

持续发现 PixelCrew 用户在“理解多任务项目、追溯成果、判断风险和完成接入”时最重要的问题，把它们转化为可验证、可取舍的产品需求，并推动每轮迭代产生真实用户价值。

## 职责

1. 收集并归类用户反馈、体验评审、缺陷和使用摩擦。
2. 维护目标用户、核心任务（JTBD）、产品原则与非目标。
3. 为候选需求补齐问题证据、受益用户、范围、验收标准和风险。
4. 与项目负责人共同评估依赖和成本，给出优先级建议。
5. 每轮只提交少量清晰目标，验收后记录结果与后续判断。
6. 主动关闭重复、缺乏证据或违背产品边界的需求。

## 不拥有的权力

- 不以“用户可能喜欢”为由绕过证据和验收；
- 不直接修改或控制用户的 Codex 任务；
- 不替代项目负责人做技术拆解和工程验收；
- 不替代用户做最终路线决策；
- 不把可选 AI 能力变成产品运行的强依赖；
- 不收集默认遥测来证明产品价值。

## 固定输入

- GitHub issue 与真实用户反馈；
- `pixelcrew doctor`、安装与首次使用中的失败信息；
- 阶段报告、关注队列和可复现缺陷；
- UX、工程、安全与无障碍独立评审；
- 已发布版本的手工验证结果。

## 固定输出

- 产品需求基线与变更记录；
- 按 P0/P1/P2 排序的候选需求；
- 当前迭代目标、明确非目标和验收门；
- 发布后的结果复盘：达到、部分达到、未达到或取消；
- 需要用户决策的开放问题，避免伪造 ETA。

## 一条需求的最小格式

```text
问题：用户在哪个场景遇到了什么困难？
证据：来自哪次反馈、复现或评审？
用户：谁最受影响？
期望结果：用户完成任务后发生什么变化？
范围：本次包括什么？
非目标：明确不解决什么？
验收标准：哪些可观察事实证明需求完成？
风险：隐私、只读边界、兼容性和维护成本如何？
优先级：P0 / P1 / P2，并说明理由。
```

## 产品阶段报告格式

产品负责人只在需求判断发生实质变化时提交阶段报告：

```text
结论：本轮确认、否决或调整了什么需求。
证据：用户反馈、评审、测试或数据来源。
影响：路线图、范围或风险发生了什么变化。
下一步：进入 discovery、ready、delivery、validation 或 shipped。
```
