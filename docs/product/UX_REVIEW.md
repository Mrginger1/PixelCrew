# PixelCrew UX / 产品体验独立评审

> 评审日期：2026-07-24
> 评审范围：`src/pixelcrew/web/index.html`、CLI 首次接入流程、当前本地项目快照、README/Quick Start/Secretary 文档，以及 dashboard / 员工报告 / checkpoint 截图。
> 评审视角：项目负责人第一次接入后，能否在 30 秒内回答“现在怎样、哪里有风险、下一步该做什么、证据在哪里”。

## 一句话结论

PixelCrew 已经有鲜明、可记忆的“像素办公室”产品形态，也完成了任务发现、员工档案、阶段记录和规则秘书的闭环雏形；但目前仍更像一张**会动的项目海报**，还不是一个负责人可以放心据此决策的**项目控制台**。最需要优先解决的不是继续加装饰，而是状态可信度、成果可达性、行动闭环和首次接入自解释能力。

## 严重度定义

- **P0 — 信任或核心任务阻断**：会让用户得出错误项目判断，或无法完成产品承诺的核心动作。
- **P1 — 主要流程显著受损**：可以绕过，但负责人理解项目、追踪员工或完成接入的成本很高。
- **P2 — 体验质量问题**：不阻断流程，但持续损害可读性、无障碍、移动体验或品牌可信度。

---

## P0 — 必须先解决

### P0-1｜看板给出“精确结论”，却没有交代状态的可信度

**现象**

- 顶部直接显示“正在推进”“需要关注”和百分比；项目观测台又给出总进度、成果证据 100%、汇报 100% 等确定性指标。
- 当前真实快照中，负责人任务的 assignment 仍停留在“等待确认后创建 GitHub 仓库”，但仓库事实上已经发布；这说明“任务最后可解析文本”不一定等同于“当前真实状态”。
- 每名员工均显示 8 个成果、10 份报告，证据雷达因此达到 100%，但缺少去重依据、来源引用和异常提示，用户无法判断这是五份独立交付，还是同一路径在多个任务中被重复识别。
- `overallProgress` 是员工进度的算术平均，而界面没有解释口径；没有计划的任务、大小悬殊的任务和已完成任务会被同权处理。

**影响**

负责人会把“算法推断”误读成“事实”。一旦用户发现一个过期 assignment 或重复成果，后续所有红黄绿灯、百分比和秘书结论都会失去信任；这是项目态势产品最严重的问题。

**建议**

1. 为每个核心结论提供 `来源 + 更新时间 + 置信度/异常`：例如“来自 5 个任务的最新结构化计划，2 项存在文本冲突”。
2. 把状态拆成三层：**原始事实**（最后更新时间、完成步骤、文件存在性）、**规则推断**（疑似等待、疑似过期）、**AI 判断**；视觉上和文案上明确区分。
3. 对 assignment 与完成事件冲突、同一路径跨员工重复、任务长期 active 但无进行中步骤等情况建立数据质量告警。
4. 总进度默认显示为“计划覆盖情况”，不要伪装成项目完成度；提供口径说明，并允许项目选择按步骤、里程碑或权重计算。

**可验证的完成标准**

- [ ] 任一顶部数字或雷达指标可点击查看计算口径、输入任务和最后更新时间。
- [ ] 人为构造“旧 assignment + 新完成记录”时，页面显示冲突提示，而不是无条件展示旧 assignment。
- [ ] 同一成果路径出现在 3 个任务中时，全局成果只计 1 份，并保留全部引用来源。
- [ ] 用户测试中，5/5 名新用户能准确说出“52% 是什么进度”，不会回答成泛化的项目完成度。

### P0-2｜“成果可追溯”目前实际只是“复制路径”

**现象**

- 首页成果柜、员工档案和报告里的成果按钮，点击后统一执行 `copyPath()`；没有预览、在 Finder 中显示、打开本地文件、打开目录、访问网页或查看远程文件的分流。
- 卡片视觉上像“可打开的成果”，但反馈只是“路径已复制”，与 README 中“Browsable / traceable evidence”的预期不一致。
- 里程碑详情只展示文字、进度和完成项，没有把“这次里程碑由哪些文件/测试/链接证明”绑定到记录本身。
- 全局成果柜只截取去重后的前 4 项，没有“查看全部”、筛选、搜索或不可用状态。

**影响**

用户知道“有成果”，却仍要离开 PixelCrew、寻找终端、粘贴路径并判断用什么程序打开。检查 checkpoint 和验收证据的核心工作没有闭环，成果柜更像装饰而不是工具。

**建议**

1. 本地 allowlist 内的成果提供明确动作：**预览、打开、在 Finder 中显示、复制路径**；URL 提供“打开链接”；远程路径提供“复制 SSH 命令/连接说明”。
2. 点击成果先打开轻量预览抽屉，展示类型、大小、修改时间、存在性、所属员工、关联里程碑和可用动作，不要让“单击”暗中等于复制。
3. 里程碑数据结构增加 `evidenceRefs`，让文件、测试结果、视频和报告与某次 checkpoint 直接关联。
4. 增加完整成果库；首页只保留“最近更新 / 待验收”快捷入口。

**可验证的完成标准**

- [ ] allowlist 内的图片、视频、Markdown/PDF、目录和 URL 各有符合类型的首要动作。
- [ ] 文件已删除或远程不可达时显示明确状态，不出现“复制成功即代表成果有效”的假象。
- [ ] 从任一里程碑详情最多 1 次点击即可看到其关联证据；返回后仍保留原员工档案位置。
- [ ] 首页存在“查看全部成果”，30 个成果均可通过员工、类型、里程碑和时间筛选。

## P1 — 下一轮核心产品迭代

### P1-1｜首屏优先展示“办公室”，而不是负责人要处理的事情

**现象**

- 1440px 桌面截图中，顶部摘要、640×390 像素办公室、花名册和单员工详情占据首屏主体；真正帮助决策的项目观测台、秘书建议、员工汇总和成果在下方。
- “自动简报”和“秘书简报”内容高度重复，占用两块视觉重量。
- 负责人进入页面后，视线先被动画人物和建筑吸引，但“今天需要我决定什么”“谁在等待谁”“哪个里程碑刚完成”没有形成首要入口。

**影响**

品牌记忆强，但日常使用效率低。用户每次打开都需要滚动和扫读，难以在 30 秒内形成项目判断。

**建议**

把默认首页改成“负责人晨会”：首屏依次为 **需要我处理（阻塞/决策/等待）→ 本轮目标与健康度 → Crew 当前动作 → 最近成果**。像素办公室保留为产品签名，可作为中部可折叠的“办公室视图”或 Home / Office 双视图之一。合并自动简报与秘书简报：同一事实摘要只出现一次，AI 只补充规则无法表达的关联判断。

**可验证的完成标准**

- [ ] 在 1366×768 下，无滚动即可看到需要关注项、当前阶段、下一步和最近一个成果。
- [ ] 页面不存在两段语义重复的项目简报。
- [ ] 5 名用户在 30 秒内能回答：当前目标、最大风险、下一位等待协调的员工、最近成果。
- [ ] 办公室视觉仍保留，但隐藏/折叠后不影响全部管理功能。

### P1-2｜首次接入依赖用户自己拼接 CLI 流程，失败后没有产品内恢复路径

**现象**

- `init` 只提示下一步运行 `doctor`；`doctor` 成功后不继续提示 `serve`，`serve` 也不自动打开浏览器。
- 配置默认写在“当前终端目录”，但命令参数是目标项目目录，新用户容易不知道之后应在哪个目录运行 `serve`。
- “发现 0 个任务”、端口占用、配置缺失、Codex 状态库找不到等问题主要依赖终端文字和文档排查；网页没有 Setup / Empty / Offline 引导。
- README 英文主入口与 CLI 中文输出不一致；安装只给 editable install，容易让非开发用户误解这是必须进入源码目录才能使用。

**影响**

第一次成功启动要靠阅读文档和理解配置文件，不符合“init + serve，无办公室搭建仪式”的承诺。安装成功不等于首次价值达成。

**建议**

提供 `pixelcrew start /path/to/project` 作为 happy path：创建或复用配置、执行 doctor、启动服务、打开浏览器，并打印停止方式。网页增加首次接入向导和可复制诊断命令；CLI 每一步都明确显示“当前完成 / 下一条命令 / 配置位置 / 访问地址”。

**可验证的完成标准**

- [ ] 一名不了解仓库结构的新用户用一条命令在 2 分钟内看到第一个 Crew。
- [ ] 配置缺失、0 任务、端口占用和 Codex 未登录都有专属错误码、原因和下一步。
- [ ] `doctor` 成功后明确给出可直接复制的 `serve --config ...` 命令。
- [ ] 从任意工作目录都能可靠启动已初始化项目，且 UI 显示当前读取的是哪个项目。

### P1-3｜“员工”与“任务”的身份模型不稳定，项目越大越难读

**现象**

- 产品文案把每个 Codex task 映射成一名员工，但任务可能是一次性评审、临时诊断、同一角色的续作，未必等于稳定团队成员。
- 默认名称、职责和头像依赖配置或解析结果；同一职责的新任务可能变成新员工，旧任务可能长期留在花名册。
- 员工档案突出 assignment、计划和最新汇报，却没有清晰显示：上级目标、依赖谁、正在等什么、验收人、任务生命周期、是否已被替代。
- 当前每层固定 6 人；超过 6 人后靠楼层切换，跨层风险不在同一视野内。

**影响**

用户会把“线程状态”误当成“人员状态”。在 10–30 个任务时，办公室迅速变成任务陈列馆，而不是可管理的组织视图。

**建议**

分离 **Crew（稳定角色）** 与 **Assignment（可结束任务）**。一个 Crew 可以承接多个任务，任务有 active / superseded / archived 生命周期；支持按工作流、状态、Owner 分组，而不只按每层 6 人分页。档案首屏加入目标、依赖、验收条件和接棒关系。

**可验证的完成标准**

- [ ] 同一角色连续创建 3 个任务时，默认归入一个 Crew 的任务历史，而不是生成 3 名员工。
- [ ] 被替代或归档任务不会计入“正在推进”，但仍可在档案历史中查看。
- [ ] 20 个任务时，可在 10 秒内筛出“阻塞、等待我、待验收”三类任务，不需要逐层查看。
- [ ] 每个 Assignment 都能回答“服务哪个目标、依赖谁、由谁验收”。

### P1-4｜里程碑浏览只有“弹窗套弹窗”，没有项目级时间线和可分享定位

**现象**

- 用户必须先打开某员工汇总卡，再在木板报告中点历史记录，才能进入 checkpoint 二级弹窗。
- 历史在 UI 中最多显示有限条目，服务端阶段报告也有截断；没有“还有 N 条”、分页、搜索、按类型筛选或跨员工时间线。
- 详情状态没有写入 URL；刷新、复制链接或浏览器后退都无法回到某员工/某里程碑。
- 嵌套 modal 虽有 Escape 分层关闭，但增加认知负担，也不适合移动端长内容。

**影响**

“最近成果”可以浏览，但历史审计、周报回顾、跨员工比较和把某条里程碑发给同事都很困难。

**建议**

新增项目级 Timeline / Reports 页面；按时间、员工、类型、状态和证据过滤。员工档案改为可路由侧栏或页面，里程碑使用独立路由（例如 `#/crew/:id/milestones/:id`），支持深链、刷新恢复和浏览器前进后退。显式显示截断与加载更多。

**可验证的完成标准**

- [ ] 用户从首页最多 2 次点击进入任一员工的任一里程碑。
- [ ] 复制当前 URL 到新标签页后，能恢复同一里程碑详情。
- [ ] 50 条历史记录不会丢失，可按员工/类型/日期/是否有证据筛选。
- [ ] 浏览器后退按“里程碑 → 员工 → 项目”顺序工作。

### P1-5｜秘书只会“播报”，还没有成为真正的协调入口

**现象**

- 秘书卡显示 briefing 和 next actions，但建议动作只是静态列表，不能跳到相关 Crew、证据或待确认问题。
- 规则秘书和 AI 秘书的差别主要体现在标签；用户看不到“这句话引用了哪些事实”“为什么建议这个动作”。
- AI 秘书由独立 CLI 命令触发，页面没有引导、刷新状态、成本提示、上次失败原因或缓存生命周期操作。
- 当前规则简报与观测台自动简报基本重复，AI 若只改写同一段文本，价值不够清晰。

**影响**

秘书具有人格但缺少工作能力。用户阅读后仍要自己定位对象、判断依据和执行下一步。

**建议**

把秘书定位成“项目导航与问答层”，不是第二份摘要。每条建议都附来源 chips，并提供“查看相关 Crew / 打开证据 / 标记已处理”。AI 模式重点回答跨任务依赖、冲突、遗漏和备选决策；规则模式负责可复现事实。允许在页面中安全地触发一次 AI 更新，但明确本次调用、脱敏范围和失败降级。

**可验证的完成标准**

- [ ] 每条建议至少链接到一个任务、里程碑或证据来源。
- [ ] 用户能区分事实、规则推断和 AI 建议，且 AI 文字不冒充状态事实。
- [ ] 页面能显示 AI 简报生成中、成功、过期、失败及回退原因。
- [ ] 从“某员工正在等待”的建议 1 次点击即可打开对应档案并看到等待对象。

### P1-6｜关键无障碍交互不完整，键盘和屏幕阅读器会卡在弹窗里

**现象**

- attention queue 使用带 click 的 `<li>`，不是 button/link，键盘无法触发。
- 报告和记录弹窗没有 focus trap；打开二级弹窗时，一级弹窗仍可被辅助技术访问；关闭后虽尝试恢复焦点，但 Tab 可进入背景页面。
- 进度条、四向雷达和同步状态大多是视觉 `div/i`，没有 `role="progressbar"`、`aria-valuenow` 或 live region。
- toast、同步成功/失败和定时刷新不会通过 `aria-live` 宣告。
- 部分卡片 `focus-visible` 只做位移或移除 outline，没有稳定的高对比焦点环。

**影响**

键盘用户无法完成关注项跳转；屏幕阅读器用户无法理解数值、刷新结果和弹窗边界，也可能迷失焦点。这不是“锦上添花”，而是核心流程不可用。

**建议**

按 WCAG 2.2 AA 修复语义、焦点和状态通知：所有可点击对象使用原生 button/link；dialog 打开时设置背景 inert 并循环焦点；进度和健康指标提供可读名称/数值；动态区域使用克制的 `aria-live`；所有操作保留可见焦点。

**可验证的完成标准**

- [ ] 仅用键盘可以完成：选择员工 → 打开报告 → 打开里程碑 → 返回原卡片。
- [ ] dialog 打开后 Tab/Shift+Tab 不会进入背景；二级 dialog 关闭后焦点回到触发记录。
- [ ] axe-core 自动扫描无 critical/serious 问题，并通过一次 VoiceOver 人工流程。
- [ ] 所有焦点态在浅色和深色背景上达到 3:1 非文本对比度。

### P1-7｜移动端是桌面布局的压缩版，不是移动任务流

**现象**

- 720px 以下仍保留完整 canvas、绝对定位人物和 speech bubble；画布缩小时，角色热区与气泡容易拥挤，信息密度失控。
- 花名册仍是横向 flex 滚动；移动 media rule 设置 `grid-template-columns` 对 flex 容器无效。
- 大量正文为 8–11px，按钮和成果 chip 的触摸目标明显小于 44×44px。
- 员工报告和里程碑是长弹窗，嵌套后返回层级不明显；没有底部导航或移动端主任务入口。

**影响**

手机上“看看项目现在怎样”本应是高频场景，但当前更适合缩放观赏，不适合快速处理风险、阅读报告或打开成果。

**建议**

移动端采用不同信息架构：默认显示 Today、Attention、Crew、Reports 四个纵向区块或底部导航；办公室画布变成可选横向预览。员工档案和里程碑改为全屏 sheet/page，最小正文 14px、次要文字 12px、触摸目标 44px。

**可验证的完成标准**

- [ ] 在 360×800 和 390×844 下无横向页面滚动，无人物/铭牌/气泡遮挡关键操作。
- [ ] 首屏无需缩放即可读到风险、下一步和更新时间。
- [ ] 所有主要触摸目标至少 44×44 CSS px，相邻目标间距至少 8px。
- [ ] iPhone Safari 与 Android Chrome 完成核心流程，弹窗打开时背景不滚动。

## P2 — 品质与一致性

### P2-1｜视觉层级仍偏“处处高声”，数据可读性让位于像素装饰

**现象**

- 深色描边、硬阴影、木板、纸张、胶带、印章、像素字体和中英双标签在几乎每个区域同时出现。
- 多数辅助信息只有 7–10px；英文大写 eyebrow 数量过多，用户需要先穿过装饰性术语才能读中文任务内容。
- 状态、可点击卡片和纯展示卡片外观相近；“成果柜卡片点击复制路径”也没有明显动作提示。

**影响**

界面有风格但扫描效率低。长时间使用时视觉疲劳，重要风险与普通装饰争夺注意力。

**建议**

保留一个品牌签名——像素办公室；其余区域降低描边和装饰密度。建立清晰字号阶梯（正文 ≥14px，辅助 ≥12px），英文 eyebrow 只用于真正的结构标签。用一致的 action affordance 区分“可打开、可筛选、仅展示”。

**可验证的完成标准**

- [ ] 负责人关注项是全页唯一最高警示层级，普通卡片不与其竞争。
- [ ] 正文和控件说明在 100% 缩放下无需放大即可连续阅读。
- [ ] 5 名用户无需点击即可区分可打开卡、静态指标和复制动作。
- [ ] 核心页面通过 200% 文本缩放，不裁切任务内容或丢失操作。

### P2-2｜“Local-first”被外部字体和持续动画削弱

**现象**

- 页面通过 Google Fonts `@import` 拉取字体；离线、国内网络或严格隐私环境中会失败并产生外部请求。
- canvas 动画使用持续帧循环；`prefers-reduced-motion` 只关闭选中 pin，没有停止人物、场景和 live pulse 动画。
- 用户没有暂停动画的控制。

**影响**

本地优先产品在首次加载时仍依赖公网，字体回退会破坏视觉；持续动画增加 CPU/电量消耗，也可能给前庭敏感用户造成不适。

**建议**

将许可允许的字体随 wheel 打包，或使用可靠系统字体栈。尊重 `prefers-reduced-motion`：停止 canvas 动作、脉冲和位移动画，并提供“暂停办公室动画”开关，标签页不可见时暂停绘制。

**可验证的完成标准**

- [ ] 断网启动时无第三方请求、无字体闪烁，页面视觉结构稳定。
- [ ] reduced motion 下 canvas 为静态帧，CSS 动画和过渡停止。
- [ ] 页面切到后台后动画循环暂停；返回后恢复且状态数据不丢失。
- [ ] Lighthouse 网络记录中仅访问 `127.0.0.1` 资源。

### P2-3｜空、错、旧三种状态缺少足够区别

**现象**

- API 失败后界面保留 fallback/旧快照，只在同步时间和页脚提示“离线快照”；顶部数字和员工卡仍可能看起来像实时数据。
- 0 任务、无计划、无报告、无成果多以一句占位文字处理，没有直接修复动作。
- AI 缓存过期会回落规则秘书，但用户不清楚“哪些内容仍可信、哪些内容已过期”。

**影响**

旧数据容易被当作 live data；新用户遇到空页面时不知道应创建任务、调整 workspace_match，还是修复权限。

**建议**

建立全局状态模型：Loading / Live / Stale / Offline / Empty / Misconfigured。Stale/Offline 时在页面顶部持续显示非模态 banner，并给所有时间敏感指标加“截至某时”；Empty 与 Misconfigured 提供可复制命令和 doctor 结果入口。

**可验证的完成标准**

- [ ] 断开服务后 2 秒内出现全局离线 banner，所有旧数据标记统一的“截至时间”。
- [ ] 0 任务页面包含原因检测和至少一个可执行下一步，而不是空办公室。
- [ ] AI 缓存过期只影响 AI 内容，规则事实仍明确标记为实时/旧快照。
- [ ] 恢复服务后 banner 自动消失，并通过 live region 宣告恢复。

### P2-4｜中英文产品体验不一致

**现象**

- 主 README 有完整英文版和中文版，但 Web UI、CLI 帮助与错误主要是中文，并混入大量英文大写标签。
- 当前混合语言更像视觉风格，而不是可选择的语言系统；英文用户无法获得完整可用体验。

**影响**

开源传播看起来国际化，实际首次运行却不是。术语也容易漂移，例如 task / crew / employee / assignment / dossier 没有稳定词汇表。

**建议**

建立 i18n 词条和术语表，至少支持 `zh-CN` / `en` 自动检测与手动切换；CLI 根据 locale 输出。品牌词可保留英文，但功能标签不要双语叠加。

**可验证的完成标准**

- [ ] Web、CLI 和错误帮助均可完整切换中文/英文，不出现同一控件双语堆叠。
- [ ] README 两种语言的命令、功能承诺和截图说明一致。
- [ ] 术语表明确 Crew、Assignment、Milestone、Evidence、Secretary 的唯一含义。
- [ ] 英文环境首次接入测试不需要阅读中文文本。

## 3 个低成本、高收益 Quick Wins

### Quick Win 1｜在最顶部增加“负责人待办条”

把“阻塞、等待用户输入、待验收、状态冲突、最新完成”压成 3–5 条可点击事项，放在办公室之前；无事项时显示“当前无需负责人介入”，而不是只显示 0。

**为什么值**：不重做数据层和办公室，就能让首页从“展示状态”升级为“指导注意力”。
**一周内验收**：1366×768 首屏可见；点击事项定位员工；0/1/5 条均有清晰状态。

### Quick Win 2｜成果点击改成小型动作菜单，而不是直接复制

第一步先支持 `打开 / Finder 中显示 / 复制路径`（本地 allowlist）和 `打开链接 / 复制`（URL）；远程路径维持复制，但明确写“远程路径不可直接打开”。

**为什么值**：直接补上用户已经反复提出的“阶段成果打不开”痛点，也让 README 的 evidence 承诺更可信。
**一周内验收**：图片、视频、目录、URL、远程路径五类各走正确动作；失败有明确提示。

### Quick Win 3｜补齐状态与弹窗的基础无障碍

把 attention `<li>` 改为 button；给 toast 增加 `role="status"`；给进度加 `role="progressbar"`；dialog 使用 `inert + focus trap`；为所有卡片补可见焦点环；同步状态增加 aria-live。

**为什么值**：改动集中、不改变视觉方向，却能修复整条键盘流程和自动化审计中的主要高危项。
**一周内验收**：键盘完成员工→报告→里程碑→返回；axe 无 critical/serious。

## 理想版体验蓝图

### 产品定位

> PixelCrew 不是把任务“画成小人”的皮肤，而是一个以证据为核心、以项目负责人注意力为首页的本地 Agent 项目操作台；像素办公室是它最有记忆点的空间隐喻。

### 信息架构

```text
┌─────────────────────────────────────────────────────────────┐
│ Project / Sync / Data confidence / Search / Language        │
├─────────────────────────────────────────────────────────────┤
│ TODAY：需要我决定 2 · 等我输入 1 · 待验收 3 · 新完成 2       │
├───────────────────────────────┬─────────────────────────────┤
│ 本轮目标与健康度              │ 秘书判断                    │
│ 目标 / 阶段 / 依赖 / 风险     │ 事实引用 / 建议 / 待确认     │
├───────────────────────────────┴─────────────────────────────┤
│ Crew lanes：按目标或工作流分组；状态、依赖、下次检查点       │
├─────────────────────────────────────────────────────────────┤
│ 最近里程碑与待验收证据                                      │
├─────────────────────────────────────────────────────────────┤
│ 可折叠 Pixel Office（品牌签名 / 空间导航 / 轻动画）           │
└─────────────────────────────────────────────────────────────┘

主导航：Today · Crew · Timeline · Evidence · Office · Settings
```

### 核心对象模型

- **Goal**：项目为什么做，本轮成功标准是什么。
- **Crew**：稳定角色或职责，可承接多个 Assignment。
- **Assignment**：一个可结束、可替代、可归档的 Codex 任务。
- **Milestone**：发生了可审计变化的阶段记录。
- **Evidence**：文件、链接、测试、视频、模型等可验证成果。
- **Decision / Attention**：需要负责人确认或协调的事项。
- **Secretary Brief**：基于上述对象的事实摘要与可解释建议，不是另一份聊天记录。

### 理想首次接入

1. `pixelcrew start /project`；自动检测 Codex、创建配置、扫描任务。
2. 浏览器打开 Setup Check：找到多少任务、准备生成哪些 Crew、哪些路径会被读取。
3. 用户确认稳定角色映射，也可以跳过使用自动建议。
4. 进入 Today，首次解释每个指标口径；示例成果和真实成果明确区分。
5. 页面给出“创建第一个可追溯里程碑”的短引导，而不是要求用户先读架构文档。

### 理想日常使用

1. 用户打开 Today，先处理“等我输入 / 待验收 / 状态冲突”。
2. 点某项打开 Crew 侧栏，看到目标、当前动作、依赖、最新汇报和证据。
3. 点里程碑进入可深链详情，直接预览视频/报告/测试结果。
4. 秘书解释跨任务冲突并引用来源；用户可标记已处理，但 PixelCrew 不擅自修改 Codex 任务。
5. Timeline 保留完整历史；周报可以按时间范围生成并导出。

### 有创意但有用的后续能力

- **Fog of Work（工作迷雾）**：不是装饰性雾效，而是把“信息过期/缺少计划/证据未验证”可视化；点击即可看到补全方式。
- **Evidence Weather（证据天气）**：天气不再是固定太阳，而由测试通过率、数据新鲜度和阻塞决定；必须同时提供文字状态，避免只靠图形。
- **Handoff Replay（交接回放）**：按时间播放目标、决策、里程碑和证据变化，帮助新负责人快速接手。
- **Decision Debt（决策债）**：识别长期无人确认的问题、反复打开的风险和没有证据的“已完成”。
- **Project Lens（项目镜头）**：同一数据切换“负责人 / 工程 / 研究 / 发布”视角，不同类型项目通过 lens 适配，而不是重写办公室。

## 推荐迭代顺序

1. **信任地基**：P0-1 状态口径与来源、P0-2 成果可达性。
2. **负责人首页**：P1-1 信息架构、Quick Win 1、消除简报重复。
3. **可持续模型**：Crew / Assignment 分离，项目级 Timeline 与深链。
4. **行动秘书**：来源引用、可点击建议、显式 AI 生命周期。
5. **体验底线**：键盘/屏幕阅读器、移动端、离线与空状态。
6. **品牌精修**：在可读性与性能达标后，再继续丰富像素办公室细节。

## 北极星验收问题

下一版不应以“画面是否更丰富”作为主要成功标准，而应以以下五问衡量：

1. 用户能否在 **30 秒**内说出项目目标、最大风险和下一步？
2. 用户能否在 **2 次点击**内从风险进入负责人、里程碑和证据？
3. 任一状态结论能否解释“**来自哪里、截至何时、可信到什么程度**”？
4. 断网、0 任务、旧缓存和文件丢失时，产品能否明确指导恢复？
5. 手机、键盘和 VoiceOver 用户能否完成同一条核心流程？
