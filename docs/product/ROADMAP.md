# PixelCrew 产品路线图

> 更新日期：2026-07-24
> 状态：Cycle 1 已验收；Cycle 2 待进入 Ready 评审
> 路线图表达当前判断，不是长期交付承诺。每轮必须经过需求准备、独立验证和结果复盘。

## 三方评审共识

产品、UX 和工程评审从不同角度得出了四个共同结论：

1. **先建立信任，再增加装饰。** 状态、进度、关注项必须说明来源、时间和推断口径。
2. **先补齐核心旅程。** 安装、启动、诊断、打开、停止目前仍需要用户理解 Python 与前台进程。
3. **成果必须真正可达。** “复制路径”不等于证据预览，需要安全、白名单内的打开和下钻闭环。
4. **先稳定数据边界，再承诺跨平台。** Codex Adapter、增量解析、快照契约和最小披露 API 是长期可迁移性的基础。

评审同时明确：现阶段不做任务写回、自动派工、公网 SaaS、Jira/Linear 替代、插件市场或更多纯装饰动画。

## 优先级决策

| 顺序 | 主题 | 为什么现在做 | 退出标准 |
|---:|---|---|---|
| 1 | 本地生命周期闭环 | 真实用户已出现“装过但不知道怎么启动、服务退出后打不开”的问题；改动集中、价值直接 | `start/status/open/stop` 可诊断、不会重复启动或误杀 Codex，文档和测试同步 |
| 2 | 版本化 Adapter 与增量快照 | 当前 5 个任务每次刷新会全量读取约 62 MB rollout，接口约 1.6–2.0 秒；也是兼容性和隐私 DTO 的依赖 | schema probe、能力状态、cursor/cache、请求合并及性能证据 |
| 3 | 状态来源与负责人首页 | UX 首要风险是“精确但不可解释”；项目负责人需要先看到待介入事项 | 状态标记 observed/derived/unknown，可下钻来源；首屏 30 秒内读懂风险与下一步 |
| 4 | 安全成果阅览闭环 | 用户已经明确反馈成果打不开；但必须先完成 opaque handle、canonical path 和 allowlist | 图片/文本/PDF/视频安全预览；目录、本地/远程路径动作明确；越界测试通过 |
| 5 | 可访问性、移动端与双语 UI | 当前品牌成立，但核心路径对键盘、VoiceOver、手机和英文用户不完整 | 核心流程键盘可达、减少动画、响应式任务流、zh-CN/en 词条完整 |
| 6 | 多项目与第二数据源验证 | 只有稳定数据契约后，迁移能力才不是口号 | 多项目严格隔离；第二个最小 Adapter 不修改领域/UI 即可接入 |

## Cycle 1 — 启动即可信 ✅

**产品问题**：用户已经安装或克隆 PixelCrew，却不知道服务是否安装、配置在哪里、是否还在运行；前台终端退出后页面直接打不开。

**本轮范围**：

- 增加 `pixelcrew start`、`status`、`open`、`stop`；
- 服务状态与日志只写入项目本地 `.pixelcrew/`；
- 重复启动、端口冲突、坏配置和进程退出有明确提示；
- 停止命令验证进程身份，只停止 PixelCrew，不触碰 Codex/Agent；
- 保留 `serve` 前台模式和已有命令兼容；
- 补齐 CLI 自动测试与中英文使用说明。

**明确非目标**：不自研通用守护进程，不安装 launchd/systemd，不做自动开机启动，不操作其他任务，不依赖 LLM。

**验收问题**：

1. 用户能否只凭命令输出知道办公室是否运行、地址和日志在哪里？
2. 终端关闭后，`start` 启动的服务是否仍可访问？
3. 重复 `start` 是否不会产生第二个隐蔽进程？
4. 端口被其他程序占用时是否给出可执行建议？
5. `stop` 是否有测试证明不会误杀 Codex 或 PID 复用后的无关进程？

**结果（2026-07-24）**：达到。26 项单元测试通过；并发启动、普通前台 SIGTERM 与健康检查阶段早期 SIGTERM 实测通过；wheel 隔离安装和源码 checkout 两条真实生命周期均完成 `init → start → status → /api/status → stop`。详细证据见 [`CYCLE_1_REVIEW.md`](CYCLE_1_REVIEW.md)。

## Cycle 2 — 数据可信且跑得久

围绕 Codex Adapter、schema/capability probe、rollout cursor、快照缓存、请求合并和 `/health` 建立稳定采集层。目标是无变化热快照 p95 < 100 ms，单 rollout 小幅追加 p95 < 300 ms，并在不兼容时显示 `OK / DEGRADED / INCOMPATIBLE`。

## Cycle 3 — 负责人 30 秒读懂项目

为状态、进度和关注项增加来源、时间与置信类型；首屏增加 3–5 条负责人待办；每项两次点击内进入员工、检查点和证据。无计划时显示“未建立计划”，不再制造精确百分比。

## Cycle 4 — 证据能打开，也守边界

设计最小披露的成果 handle 与服务端 allowlist 校验，然后提供 `打开 / Finder 中显示 / 复制路径` 及安全预览。远程路径保持明确的不可直接打开状态。目录穿越、symlink 逃逸、编码路径和未知 MIME 必须先有拒绝测试。

## 成功指标

**北极星**：项目负责人能否在 3 分钟内形成一个有来源、有证据的下一步决策。

辅助指标：

- 首次接入成功率及失败原因可诊断率；
- 30 秒内识别项目目标、最大风险和下一步的任务成功率；
- 状态结论具有来源说明的覆盖率；
- 完成员工中具有可达证据的覆盖率；
- 热刷新 p95 与单次增量刷新 p95；
- 核心流程键盘、移动端和双语完成率。

PixelCrew 默认不采集遥测；上述指标通过发布前可复现实验、可选本地诊断和用户主动反馈验证。
