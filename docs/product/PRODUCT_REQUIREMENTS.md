# PixelCrew 产品需求文档（PRD）

> **文档状态**：Draft v1.0
> **适用版本**：基于 PixelCrew v0.3.0 现状评审
> **产品边界**：Local-first · Read-only · 默认不依赖 LLM
> **证据范围**：README、现有 docs、`pixelcrew.example.json`、CLI、采集/秘书/服务端、Web UI 与测试。本文不把尚未验证的用户偏好写成事实。

## 1. 产品定义与目标

PixelCrew 是面向多任务 Agent 项目的本地项目态势工作台：把一个任务映射为一名像素办公室员工，将计划、状态、阶段报告和交付物组织成可追溯的项目视图，帮助项目负责人更快回答：**谁在做什么、主线走到哪里、哪里需要介入、结论有什么证据。**

下一阶段目标不是把它做成通用项目管理平台，而是把“从本地 Codex 项目到可信项目总览”的闭环做稳：

1. 首次使用足够简单，服务可诊断、可恢复；
2. 状态结论可解释、可回到原始证据；
3. Codex 数据格式变化时能明确降级，而不是静默误导；
4. 无网络、无模型、无账号时仍完整提供核心价值；
5. 通过稳定适配器边界支持未来迁移到其他 Agent/任务系统。

## 2. 目标用户与 JTBD

### 2.1 核心用户

| 用户 | 场景 | JTBD（待办任务） | 成功信号 |
|---|---|---|---|
| 多任务项目负责人 / PM | 同时委派多个 Codex 任务 | 当我检查项目时，我想在 3 分钟内识别主线、阻塞、负责人和证据，以便决定下一步协调动作 | 无需逐个翻聊天即可形成决策 |
| 独立开发者 / 研究者 | 多个 Agent 并行做代码、实验、报告 | 当任务运行时间较长时，我想确认真实推进而非“消息很热闹”，以便及时纠偏 | 计划、检查点、产物相互印证 |
| 团队技术负责人 | 复核阶段成果 | 当成员声明完成时，我想打开阶段记录和产物预览，以便快速验收 | 结论可追溯，缺证据会被提示 |
| 开源项目接入者 | 把 PixelCrew 用到新项目 | 当我首次接入时，我想少配置、快速诊断，以便确定它是否兼容我的环境 | 3 分钟内看到首个可信办公室 |
| 适配器贡献者（次级） | 接入非 Codex 数据源 | 当我新增数据源时，我想遵循稳定快照契约，以便不重写 UI 和秘书 | 用契约测试即可完成适配 |

### 2.2 非核心用户

- 需要多人云协作、权限审批、任务写回的组织；
- 需要 PixelCrew 自动创建、暂停、终止 Agent 的调度者；
- 需要企业级远程托管、公网分享与审计平台的用户。

## 3. 产品原则

1. **可信先于热闹**：不以消息数、动画或主观文案代表进度；状态必须有结构化数据或证据来源。
2. **本地优先**：默认只读取本机数据、监听 `127.0.0.1`，核心链路不需要云服务。
3. **只读边界**：不写 Codex 状态库，不控制任务生命周期；PixelCrew 自身配置与缓存必须与源任务状态隔离。
4. **确定性默认**：规则秘书、聚合、筛选与风险提示无需 LLM；AI 秘书仅显式启用且随时可降级。
5. **证据可追溯**：汇总必须能下钻至检查点、计划步骤和允许范围内的交付物。
6. **诚实降级**：数据缺失、解析不兼容和过期必须显式展示，不以推测填空。
7. **渐进式复杂度**：默认配置可用，高级角色映射、过滤、AI 和适配器按需启用。
8. **趣味服务于理解**：像素办公室强化空间记忆与状态感知，但不能牺牲可读性、可访问性和信息密度。

## 4. 核心使用旅程

### J1：首次接入

1. 用户安装 PixelCrew，在项目根目录运行初始化/启动命令；
2. 系统自动定位 Codex 本地状态，进行兼容性、权限和发现范围检查；
3. 若发现任务，浏览器打开办公室并解释数据来源与刷新时间；
4. 若没有任务或解析失败，给出可复制的诊断和修复建议；
5. 用户无需启用 AI 即可看到人员、计划、风险和证据。

**完成标准**：从命令执行到可信页面不超过 3 分钟，失败时也能在 1 次诊断中知道原因。

### J2：日常项目巡检

1. 用户打开总览，先看主线进度、阻塞/陈旧项与信息新鲜度；
2. 点击员工查看其阶段汇总，而非浏览大量检查点卡片；
3. 下钻某个里程碑，查看当时的完成步骤、当前步骤、进度变化和时间；
4. 返回项目层决定“继续等待、要求补证据、协调依赖或调整计划”。

**完成标准**：3 分钟内形成一个有证据支持的下一步动作。

### J3：成果验收

1. 从员工档案或成果柜打开交付物；
2. 对安全且允许的本地图片、视频、文本/Markdown/PDF进行只读预览；
3. 对模型、数据、目录或远程路径显示元信息并支持复制路径；
4. 路径不存在、越界或类型不安全时明确提示，绝不执行文件。

**完成标准**：用户可从“已完成”回溯到至少一项有效证据，或被明确告知证据缺失。

### J4：跨项目迁移

1. 对新项目运行初始化；
2. 自动建立独立配置、缓存和成果白名单；
3. `doctor` 验证发现范围和适配器能力；
4. 在不复制旧任务 ID/状态的前提下复用角色与工作架构模板。

**完成标准**：两个项目互不串数据，且无需修改前端代码。

### J5：可选 AI 秘书

1. 用户先预览脱敏后的输入；
2. 显式执行单次或值班模式；
3. 页面展示 AI 生成时间、缓存新鲜度和降级状态；
4. AI 失败时规则秘书继续工作。

**完成标准**：关闭 AI、断网或模型失败不影响 J1–J4。

## 5. 现有能力盘点（v0.4.0）

### 已具备

- `init / doctor / snapshot / start / status / open / stop / serve / secretary` CLI；
- 基于项目路径、归档状态、标题关键词/排除正则的 Codex 任务自动发现；
- 以 SQLite `mode=ro` 读取状态库，并解析本地 rollout JSONL；
- 任务状态推断、结构化计划进度、语言泡泡、阶段报告、员工汇总档案；
- 项目层计划推进、成果覆盖、报告覆盖、新鲜度与关注队列；
- 允许根目录、远程前缀、忽略目录和数量上限控制的产物识别；
- 默认规则秘书，以及显式、临时、只读、脱敏、有缓存和降级的 AI 秘书；
- 本地只读 HTTP 服务、带身份校验的后台生命周期、60 秒自动刷新、手动刷新、像素办公室和多楼层展示；
- 员工档案、二级检查点详情、成果路径复制；
- 零第三方运行时依赖、Python 3.10+、中英文 README 和基础单元测试。

### 部分具备

- **成果查看**：能识别和复制路径，但尚无安全内嵌预览、存在性/可读性反馈与下钻深链；
- **兼容性诊断**：能检查文件和任务数，但未验证数据库 schema、rollout 事件能力和快照契约版本；
- **项目迁移**：可为不同项目生成配置，但需分别启动端口，暂无统一项目切换与隔离状态提示；
- **阶段历史**：可从 rollout 重建，但受每任务最近 10 条等截断影响，也没有稳定历史索引；
- **双语**：文档双语，产品 UI 主要为中文硬编码；
- **可访问性**：弹窗具备 Escape 与焦点恢复等基础行为，但地图/Canvas 信息尚缺少完整键盘与文本等价视图。

### 尚未具备

- 明确版本化的数据源适配器与快照 schema；
- 安全的成果预览服务和稳定深链接；
- 本地历史趋势、决策时间线与版本差异解释；
- 多项目入口、项目健康摘要导出；
- 完整 i18n、无障碍标准和大规模任务性能基线。

## 6. 用户痛点与证据

| 痛点 | 仓库内证据 | 产品影响 |
|---|---|---|
| 启动链路碎片化，后台服务状态不透明 | v0.3 用户反馈与 Cycle 1 复现；v0.4 已加入 `start/status/open/stop` 并完成隔离环境生命周期验证 | 已缓解；继续观察首次接入失败是否仍集中在 Python 安装与配置位置 |
| “有成果”不等于“能验收” | Web `renderArtifacts()` 点击行为仅 `copyPath()`；API 只返回路径元数据 | 用户仍需离开产品寻找文件，验收链路中断 |
| 数据兼容错误可能被误判为无任务 | ARCHITECTURE 明确 Codex SQLite/rollout 是内部实现细节；`doctor` 主要检查文件存在和任务数量 | 数据格式变化时可能静默空字段或给出错误项目判断 |
| 状态推断缺少可解释性 | `infer_state()` 依赖更新时间、关键词和计划状态；UI 仅展示结果标签 | 用户不知道为何被判为阻塞、等待或陈旧，难以信任/纠错 |
| 历史报告可能不完整 | `stageReports[-10:]`、项目报告 `[:40]`、成果数量上限 | 长项目的关键决策可能从视图消失，阶段复盘不稳定 |
| 高级配置暴露内部任务 ID | `roles` 与 `manager_thread_id` 以 Codex thread ID 映射 | 迁移不便，也增加配置误操作和隐私泄露风险 |
| 文档国际化与 UI 不一致 | README 中英文双语；Web 文案主要为中文硬编码 | 英文用户能安装但难以完整使用 |
| 固定刷新与大项目行为未定义 | Web 固定每 60 秒全量请求 `/api/status`；每次快照重新读取/解析任务 | 任务量或 rollout 增大后延迟和资源占用不可预测 |
| 缺少稳定的适配扩展契约 | ARCHITECTURE 提到 `Dashboard.snapshot()` 是边界，但没有 schema、版本与契约测试 | 外部贡献者难以安全接入其他 Agent 系统 |
| 缺乏产品效果测量闭环 | 当前指标是项目状态指标，没有产品使用成效定义；且产品不应默认遥测 | 无法判断 PixelCrew 是否真的缩短巡检/验收时间 |

## 7. 功能需求与验收标准

### P0：可信核心闭环

#### FR-P0-1 一体化本地启动与诊断

提供清晰的本地生命周期命令（可采用 `pixelcrew start/status/open/stop`，具体 CLI 命名可在设计阶段确认），保留现有 `serve` 兼容性。

**验收标准**：

- `start` 在配置缺失时可引导初始化，在配置有效时启动服务并输出 URL、PID/进程状态和日志位置；
- 重复启动同项目不会产生不可见的重复进程；端口冲突时明确给出占用端口和可执行建议；
- `status` 能区分运行中、已退出、配置错误、数据源不可用；
- `open` 仅打开本地 URL；`stop` 只停止 PixelCrew 自身进程，不影响任何 Codex/Agent 任务；
- 支持前台模式，自动化环境不强制打开浏览器；
- 全流程不需要 LLM，不访问公网。

#### FR-P0-2 数据源兼容性与健康解释

为 Codex 采集器定义能力检查，并在 UI/CLI 展示数据是否完整。

**验收标准**：

- `doctor` 检查数据库只读打开、必要表/字段、session index、rollout 可读性与关键事件类型；
- 输出 `OK / DEGRADED / INCOMPATIBLE`，每项包含原因和建议；
- 单个 rollout 损坏不会导致整个 `/api/status` 失败；
- 页面显示数据源状态、最近成功采集时间和缺失能力（如“计划不可用，仍可显示任务”）；
- 使用只读打开方式，测试证明不会创建或修改源数据库、rollout 与任务记录；
- 为已支持的 Codex 数据样本建立回归 fixture。

#### FR-P0-3 状态来源与置信说明

每个状态、进度和关注项必须可解释。

**验收标准**：

- 员工档案可查看状态判定依据，如“计划有 in_progress”“24h 未更新”“文本命中等待资源”；
- 明确区分 `observed`（源数据直接给出）、`derived`（规则推断）、`unknown`；
- 无计划时不再用看似精确的默认百分比误导用户，UI 应显示“未建立计划”或低置信估计；
- 状态规则有单元测试覆盖边界时间、冲突信号、空数据和多语言关键词；
- 所有推断使用确定性规则，默认不调用模型。

#### FR-P0-4 安全成果预览与证据闭环

在成果白名单内提供用户主动触发的只读预览。

**验收标准**：

- 图片、文本/Markdown、PDF、浏览器支持的视频可在页面内预览；目录、模型和数据默认仅显示元信息及复制路径；
- 服务端在每次请求时重新进行规范化路径和 `allowed_roots` 校验，阻止 `..`、符号链接逃逸和未授权绝对路径；
- 不执行脚本、二进制或文档宏，不对未知类型做 MIME 猜测执行；
- 文件不存在、权限不足、远程路径不可达时显示明确状态；
- 每个成果可从员工汇总和具体检查点进入，并可返回原上下文；
- 默认不扫描白名单外目录，不上传文件，不生成公网链接。

#### FR-P0-5 版本化快照契约

把当前隐式 `Dashboard.snapshot()` 变成可测试的产品边界。

**验收标准**：

- 快照包含 `schemaVersion`、`sourceAdapter`、`capabilities` 和生成时间；
- 提供 JSON Schema 或等价机器可读契约，字段含义、可空性和枚举有文档；
- 前端对未知可选字段容错，对不兼容主版本给出可理解错误；
- Codex 适配器与 demo 数据都通过同一契约测试；
- 快照不得默认暴露源 task/thread ID、项目根绝对路径或白名单外路径。

### P1：提高决策深度与迁移能力

#### FR-P1-1 稳定的员工汇总与历史索引

**验收标准**：

- 默认仍保持“一名员工一张汇总卡”，检查点按需展开；
- 里程碑、决策、风险变化可按类型和时间筛选；
- 不因 UI 数量上限丢失源历史；若为性能分页，必须显示总数和加载更多；
- 同一 rollout 重复解析不会制造重复记录；
- 本地索引只能写入 `.pixelcrew/`，可删除重建，不修改源任务。

#### FR-P1-2 项目级决策台

**验收标准**：

- 汇总关键路径、跨任务等待、完成但缺证据、陈旧任务和最近决策；
- 每条提醒必须能下钻到员工/检查点/证据；
- 不预测未经数据支持的 ETA，不自动声称依赖关系；
- 用户可将提醒标为“已查看/误报”，状态仅保存在项目本地 PixelCrew 数据中；
- 规则秘书提供完整功能，AI 只能增强措辞和跨任务归纳。

#### FR-P1-3 多项目工作区

**验收标准**：

- 一个本地入口可列出多个显式注册的项目并切换；
- 项目配置、缓存、产物白名单、秘书状态严格隔离；
- 页面始终显示当前项目名称和根目录的脱敏标识，防止误读；
- 删除注册只移除 PixelCrew 引用，不删除项目文件或任务；
- 单项目 CLI 与配置继续兼容。

#### FR-P1-4 角色与负责人便携映射

**验收标准**：

- 默认无需用户查找 thread ID；支持按任务标题规则、项目内稳定别名或显式本地选择映射角色；
- 原始 ID 不出现在默认 UI、导出和可提交配置中；
- 匹配冲突必须提示，不静默分配负责人；
- 迁移模板不复制真实任务 ID。

#### FR-P1-5 国际化与可访问文本视图

**验收标准**：

- 至少支持简体中文和英文 UI，默认跟随浏览器且可手动切换；
- 状态、空态、诊断、秘书、弹窗和错误文案均纳入词条；
- 所有项目信息可通过非 Canvas 的文本/列表视图访问；
- 核心操作支持键盘、可见焦点、语义标签，并达到 WCAG 2.1 AA 的颜色对比要求；
- `prefers-reduced-motion` 下可关闭非必要动画。

#### FR-P1-6 本地导出负责人简报

**验收标准**：

- 可导出脱敏 Markdown/JSON 项目简报，包含生成时间、状态来源、风险和证据文件名；
- 导出前可预览将包含的字段；
- 默认不包含绝对路径、任务 ID、缓存密钥或原始聊天全文；
- 导出仅写入用户明确指定的本地路径，不自动外发；
- 无 AI 时可生成完整规则版简报。

### P2：生态与体验增强

#### FR-P2-1 通用适配器 SDK

- 提供 Adapter 协议、样例适配器、fixture、契约测试和迁移文档；
- 新适配器只能声明自己的读取范围，默认禁止写回；
- 一个非 Codex demo 适配器无需修改前端即可展示完整核心字段。

#### FR-P2-2 可配置刷新与增量解析

- 支持合理范围的刷新间隔、手动刷新和暂停；
- 只重解析发生变化的 rollout，本地缓存可安全删除重建；
- 页面展示快照耗时和最后成功刷新；
- 在 100 个任务、每个 20 MB rollout 的基准样本上，增量刷新目标 p95 ≤ 2 秒（基准机器需在测试报告中声明）。

#### FR-P2-3 视觉个性化但不破坏信息架构

- 提供有限主题、办公室布局和角色外观选项；
- 个性化配置与项目状态分离；
- 任意主题都必须通过文字溢出、重叠、缩放和对比度视觉回归测试；
- 信息层级、状态语义和键盘路径不随主题改变。

## 8. 非功能需求

### 隐私与安全

- 默认零网络请求、零遥测、零 LLM 调用；任何外部通信必须显式启用并可预览内容；
- 默认绑定 `127.0.0.1`。绑定非回环地址时必须显示高风险警告，且不将其宣传为安全公网部署；
- Codex 数据库必须以只读模式打开；所有源文件读取最小化，错误日志默认脱敏；
- 成果预览遵循白名单、路径规范化、大小限制、MIME 白名单与安全响应头；
- AI 秘书继续使用临时只读会话、脱敏、缓存过期和规则降级机制。

### 可靠性与降级

- 单任务解析失败不影响其他任务；页面显示部分失败数量；
- AI、session index、部分 rollout、单个成果不可用时核心总览仍可运行；
- API 错误不得返回本机堆栈、密钥或白名单外路径；
- 配置写入和 PixelCrew 自有缓存采用原子写，源数据始终不写。

### 性能

- 20 个常规任务的冷快照目标 p95 ≤ 1 秒，页面首个有意义内容目标 ≤ 2 秒；
- 60 秒自动刷新不得阻塞 UI 动画和交互；
- 大文件只读取必要元数据或按上限流式预览，不把二进制整体载入 JSON。

### 兼容性与可维护性

- Python 3.10+；macOS 为首要支持平台，Linux 为第二支持平台；
- 现有 CLI 和 v0.2 兼容入口在一个明确弃用周期内可用；
- 核心规则、契约、路径安全、降级和 CLI 生命周期必须有自动测试；
- 运行时继续优先零第三方依赖；如引入依赖，需说明安全、体积和维护收益。

### 可用性与可访问性

- 用户不阅读架构文档也能完成初始化、启动、诊断和停止；
- 状态色必须同时有文字/图形冗余；
- 支持 320px–1440px 关键宽度，不出现人物、铭牌和关键信息重叠；
- 动画不可遮挡内容，并支持减少动态效果。

## 9. 指标体系

PixelCrew 默认不做远程遥测。指标通过本地、可选、透明的统计或定期用户研究评估；不开启统计不影响功能。

### 北极星指标

**决策就绪巡检率（Decision-ready Review Rate, DRR）**：用户打开一个真实项目后，在 3 分钟内能够正确指出“当前主线、最高优先风险/阻塞、责任任务、至少一项支撑证据”，并形成下一步动作的巡检占比。

- Beta 目标：DRR ≥ 70%；
- 稳定版目标：DRR ≥ 85%；
- 评估方式：可选本地巡检清单 + 访谈/可用性测试，不默认上传行为数据。

### 辅助指标

| 指标 | 定义 | 目标 |
|---|---|---|
| Time to First Trusted Office | 从首次命令到看到至少 1 个正确任务及数据健康状态 | 中位数 ≤ 3 分钟 |
| 启动成功率 | 在支持环境中一次 `start` 成功或给出可执行诊断的比例 | ≥ 95% |
| 任务发现准确率 | fixture/人工标注项目中正确纳入和排除任务的比例 | Precision/Recall ≥ 95% |
| 证据可达率 | 被识别成果中可在两次点击内预览或明确解释不可预览的比例 | ≥ 95% |
| 状态可解释覆盖率 | 展示状态中具有来源类别和判定依据的比例 | 100% |
| 无模型核心可用率 | 关闭网络和 AI 时 J1–J4 全部通过的版本比例 | 100% |
| 部分故障存活率 | 一个损坏 rollout 不导致项目快照失败 | 100% 回归通过 |
| 阶段报告覆盖率 | 活跃任务中至少有一条可追溯阶段记录的比例 | 产品内显示，不强设虚假目标 |
| 误报处理率 | 被用户标记误报的关注项占比 | 用于规则改进，避免追求越低越好而隐藏提醒 |

## 10. 优先级与建议路线图

### P0：可信可用（下一里程碑）

1. 一体化启动、状态、停止与故障诊断；
2. 数据源 schema/能力健康检查与部分失败隔离；
3. 状态来源、置信和“未知”表达；
4. 成果安全预览及检查点到证据的闭环；
5. 版本化快照契约与隐私字段约束。

**发布门**：无 LLM、断网条件下完成 J1–J4；所有 P0 验收标准和安全测试通过。

### P1：决策与迁移

- 历史索引、项目决策台、多项目入口、便携角色映射；
- 中英文 UI、文本等价视图、可访问性；
- 本地脱敏简报导出。

### P2：生态与规模

- 通用适配器 SDK、增量解析与性能基准、主题个性化。

## 11. 明确不做事项

- 不创建、暂停、继续、归档或终止 Codex/Agent 任务；
- 不写回任务计划、状态、聊天记录或源数据库；
- 不做 Jira/Linear/Trello 的完整替代品；
- 不默认调用 LLM，也不把 LLM 结论当事实来源；
- 不提供未经认证的公网部署、一键公开分享或云同步；
- 不以消息数量、在线时长、人物动画频率衡量生产力；
- 不生成无依据的完成时间、依赖关系、风险或“员工绩效”；
- 不执行成果文件、脚本、模型或二进制；
- 不在近期支持移动端原生应用、多人实时编辑和企业 RBAC；
- 不为追求视觉效果牺牲信息可读性和无障碍。

## 12. 开放问题

1. “只读”是否允许用户主动调用 Finder/系统应用打开白名单文件，还是仅限浏览器内预览与复制路径？
2. PixelCrew 自有历史索引应默认启用还是按需启用？保存周期和最大磁盘占用是多少？
3. Codex 本地数据没有稳定公开契约时，首个正式支持矩阵应按 Codex 版本、schema 指纹还是能力检测维护？
4. 多项目入口应由单一常驻服务管理，还是维持每项目独立进程并提供轻量 launcher？
5. 角色映射的稳定主键应优先使用本地别名、标题规则还是由采集器提供的脱敏哈希？
6. 成果预览首批格式及大小上限如何设定，尤其是超大视频、模型和数据集？
7. 阶段报告是完全从 rollout 重建，还是允许 Agent 额外生成项目内标准报告文件供索引？
8. “依赖”只有显式声明才展示，还是可提供标记为“建议/低置信”的规则推断？建议首版仅展示显式依赖。
9. 本地可选指标如何做到透明、可删除、可导出，同时避免制造额外配置负担？
10. macOS 之外，Linux/Windows 的 Codex 数据位置和本地文件打开体验何时进入正式支持范围？

## 13. 最重要的五条产品建议

1. **先把启动与诊断做成一个可靠入口**：它直接决定开源用户能否跨过第一次体验门槛。
2. **把状态从“漂亮标签”升级为“可解释事实”**：来源、规则、时间和未知状态应随时可查。
3. **优先打通成果预览闭环**：从员工 → 检查点 → 安全预览，是 PixelCrew 区别于聊天列表和普通看板的核心价值。
4. **尽快版本化快照与适配器契约**：Codex 内部格式变化是当前最大的产品可靠性风险，也是跨项目/跨 Agent 扩展的前提。
5. **以 3 分钟决策就绪为衡量标准**：视觉、秘书、历史和指标都应服务于更快做出有证据的项目决策，而不是堆叠信息。
