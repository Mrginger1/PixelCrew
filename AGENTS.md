# PixelCrew 工作协议

本仓库使用“一个总负责人 + 多个边界清晰的执行任务 + 独立验收”的工作方式。

## 产品负责人

产品负责人维护“为什么做、为谁做、做到什么程度”，是用户需求与交付计划之间的责任角色：

1. 从真实反馈、缺陷和独立评审中维护问题清单，不把解决方案当成需求本身。
2. 为候选需求补齐目标用户、证据、范围、非目标、验收标准与隐私边界。
3. 与总负责人共同排定 P0/P1/P2；产品负责人建议价值优先级，总负责人确认交付依赖和验收路径。
4. 每轮只选择少量结果目标，发布后记录达到、部分达到、未达到或取消。
5. 不替代秘书做事实汇总，不替代总负责人做技术拆解，也不强制把可选 AI 变成核心依赖。

完整章程与状态机见 `docs/product/PRODUCT_MANAGER.md` 和 `docs/product/ITERATION_PROCESS.md`。

## 总负责人

总负责人是唯一维护总目标、里程碑、依赖关系和最终验收结论的任务。职责：

1. 把用户目标写成可验证的 Project Charter。
2. 将工作拆成互不重叠、可独立交付的工作包。
3. 只把非阻塞、边界清晰的工作包交给其他任务；自己继续关键路径。
4. 维护总计划，合并结论，处理跨任务冲突。
5. 不以“子任务说完成了”代替验收；必须复核证据。

## 执行成员

每个执行任务只拥有一个工作包，并在开始前明确：目标、输入、写入范围、依赖、验收标准、禁止事项。执行成员：

- 不扩张任务范围；新问题先记录，再由负责人决定是否拆新任务。
- 不重复其他成员的工作。
- 每次状态变化更新结构化计划。
- 交付时提供：改了什么、文件路径、验证命令与结果、风险、下一步建议。

## 验证成员

高风险或关键交付使用独立验证任务。验证成员不重做实现，只验证：可复现性、失败门槛、回归风险、验收标准是否真正满足。

## 状态语义

- `pending`：尚未开始。
- `in_progress`：正在做，并且当前动作明确。
- `completed`：成果和验证证据都已提交。
- `blocked`：存在具体阻塞条件，需要负责人或用户动作。
- `waiting`：安全等待外部资源，不应伪装成工作中。

## 强制交接格式

```text
状态：completed | blocked | waiting
完成：
- ...
交付物：
- /absolute/path/to/file
验证：
- command → result
风险/未完成：
- ...
负责人下一步：
- ...
```

## 安全边界

看板同步器只读 Codex 本地记录；不得修改任务、启动训练或执行交付物。公开仓库不得提交真实任务文本、本地绝对路径、令牌、W&B key 或私有数据。

## 阶段报告协议

每个执行任务在出现“可向用户展示的新事实”时，必须调用 `update_plan` 并在 `explanation` 中写一段阶段总结。触发点包括：完成一个里程碑、得到关键验证结果、改变技术路线、发现风险或进入等待。

阶段总结写清四件事，建议控制在 2–4 句：

1. **结论**：这阶段得到什么新结果；
2. **证据**：对应文件、测试、视频、模型或指标；
3. **影响**：它让总项目更接近完成，还是暴露了风险；
4. **下一步**：当前转入哪个计划步骤。

不要按时间机械刷报告，也不要把普通操作日志包装成成果。PixelCrew 会将这些有解释的计划变化自动归入“最近阶段成果”，并区分里程碑、技术决策和普通检查点。

## 项目秘书协议

秘书是观察与归纳角色，不是隐藏的项目负责人：

- 规则秘书可持续工作，AI 秘书必须由用户显式运行；
- 秘书只总结已有事实，不代替负责人验收、排期或下达不可逆操作；
- 跨任务简报应优先指出依赖、阻塞、证据缺口和需要用户决定的问题；
- 信息不足时明确写“不足”，不得推测 ETA、指标或成果；
- 模型输入先脱敏，输出必须显示来源和生成时间；
- AI 不可用时立即降级，不应阻断办公室同步。
